## [ALTO] - Estado derivado en dashboard con `setState` dentro de `useEffect` rompe la regla de hooks y puede causar renders en cascada
- **Archivo:** src/app/(app)/dashboard/page.tsx (línea 71)
- **Problema:** el estado `isLoggingComplete` se sincroniza desde `dashboardData` mediante un `useEffect` que llama a `setIsLoggingComplete` en cada cambio. ESLint marca además dependencia incompleta en la línea 75 (`dashboardData` no está completo en el array de deps).
- **Impacto:** añade una capa de estado duplicado respecto a React Query, abre la puerta a renders extra y deja el componente en un estado frágil para futuros cambios. Ahora mismo ya rompe `npm run lint`.
- **Fix:** eliminar el efecto y derivar el valor desde la query o encapsular el override optimista en una mutación/local reducer. Ejemplo:

```tsx
const serverIsLoggingComplete = dashboardData?.isLoggingComplete ?? false
const [optimisticLoggingComplete, setOptimisticLoggingComplete] = useState<boolean | null>(null)
const isLoggingComplete = optimisticLoggingComplete ?? serverIsLoggingComplete
```

## [ALTO] - Supabase no usa un tipo `Database` generado y obliga a casts/manual typing en queries críticas
- **Archivo:** src/lib/supabase/client.ts (línea 4)
- **Problema:** `createClient()` y `src/lib/supabase/server.ts` crean clientes sin genérico `Database`. Además `src/types/database.ts` (línea 1) es una colección manual de interfaces parciales, no el tipo generado oficial de Supabase. El resultado ya aparece en varios puntos: `useTdeeState` usa `Record<string, any>`, `useWeeklySnapshots` idem, `useDashboardData` castea `entries as FoodLogEntry[]`, `useProfile` castea `data as UserProfile`, y en `dashboard/add/page.tsx` hay callbacks inferidos como `PostgrestSingleResponse<any[]>`.
- **Impacto:** cualquier drift entre esquema real y tipos locales se convierte en bugs silenciosos en runtime: columnas renombradas, joins mal tipados, nullabilidad incorrecta y callbacks contextualizados como `any`.
- **Fix:** generar y usar el tipo oficial de Supabase, y parametrizar ambos clientes.

```bash
npx supabase gen types typescript --project-id goanhievycvklsvnsbks > src/types/supabase.generated.ts
```

```ts
import type { Database } from '@/types/supabase.generated'

export function createClient() {
  return createBrowserClient<Database>(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  )
}
```

## [ALTO] - Varias pantallas críticas ignoran `error` de Supabase y convierten fallos reales en “listas vacías” o estados incorrectos
- **Archivo:** src/app/(app)/plans/[planId]/page.tsx (línea 86)
- **Problema:** la carga inicial del detalle de plan hace `Promise.all` con cuatro queries y luego consume solo `.data`; no evalúa `planRes.error`, `daysRes.error`, `profileRes.error` ni `userPlanRes.error`. El mismo patrón aparece en `src/app/(app)/plans/page.tsx` líneas 47-56, `src/app/(app)/stats/page.tsx` líneas 49-64 y el fetch de comidas del día en `src/app/(app)/plans/[planId]/page.tsx` líneas 111-118.
- **Impacto:** errores de red, permisos o schema drift se ven como “sin datos” en vez de como fallo real. Eso degrada UX, complica soporte y puede ocultar regresiones importantes en premium/planes.
- **Fix:** propagar y mostrar un `error state` explícito por pantalla.

```ts
const firstError = planRes.error ?? daysRes.error ?? profileRes.error ?? userPlanRes.error
if (firstError) {
  setScreenError(firstError.message)
  setLoading(false)
  return
}
```

## [ALTO] - Fallos al recuperar el perfil se degradan a navegación incorrecta o estado parcial en lugar de tratarse como error
- **Archivo:** src/app/(app)/layout.tsx (línea 38)
- **Problema:** si la query a `user_profiles` falla, el layout solo hace `console.error` (línea 47), sigue con `setProfile(profile ?? null)` y redirige a onboarding si `profile` es null (líneas 50-55). En la raíz, `src/app/page.tsx` líneas 14-24 tampoco maneja `error` en la lectura de `user_profiles` y cae a onboarding por defecto.
- **Impacto:** un error transitorio de base de datos se convierte en una redirección equivocada, con riesgo de romper el flujo del usuario autenticado y generar diagnósticos falsos de “perfil incompleto”.
- **Fix:** tratar `profileError` como error bloqueante y separar “no hay perfil” de “falló la query”.

```ts
if (profileError) {
  setLoading(false)
  router.replace('/dashboard?error=profile-load')
  return
}
```

## [MEDIO] - Onboarding mezcla promesas encadenadas con efectos desactivando `exhaustive-deps`, con riesgo de closures obsoletos y doble ejecución
- **Archivo:** src/app/onboarding/page.tsx (línea 74)
- **Problema:** el onboarding usa efectos con `.then(...)` y sin manejo de error/cancelación real (líneas 74-87), y además desactiva dependencias de hooks en dos sitios sensibles: auto-detección de país (línea 96) y el flujo OAuth/auth listener (líneas 406-448). Ese segundo efecto captura `submitOnboardingData`, `clearOAuthParams` y setters sin declararlos en deps.
- **Impacto:** es fácil introducir bugs intermitentes: dobles submits, estado obsoleto al volver a onboarding, listeners que actúan con closures viejas y rutas de OAuth difíciles de depurar.
- **Fix:** extraer esos efectos a hooks dedicados con `async/await`, cancelación explícita y dependencias reales. Usar `useCallback` para las funciones capturadas por el efecto OAuth.

- **Archivo:** src/components/stats/TdeeAdjustmentCard.tsx (línea 21)
- **Problema:** varios flujos capturan errores y los ocultan:
